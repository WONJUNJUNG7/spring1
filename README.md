# spring1

1. 레이어드 계층 구조 
소프트웨어 개발에서 가장 널리 쓰이는 전통적인 아키텍처 스타일이다
시스템을 서로 다른 역할을 하는 물리적/논리적 층(Layer)으로 쪼개어 쌓아 올린 구조이다. 
표준적으로 컨트롤러, 비즈니스 논리, 데이터 액세스, 데이터베이스 계층으로 나누는데 이를 컨트롤러, 서비스, 레포지토리, 엔티티와 DTO로 나눈것이다
1. Controller (프레젠테이션 계층) : 외부에서 들어오는 요청을 가장 먼저 맞이하고 응답을 돌려주는 문지기 역할이다. 핵심 기능은 Service에게 호출하고, Service가 준 결과를 받아 클라이언트에게 응답으로 반환한다.

2. Service (비즈니스 논리 계층) : 프로그램의 핵심 규칙과 실제 기능이 수행되는 심장부이다. 핵심 기능은 필요한 데이터를 가져오거나 저장하기 위해 Repository를 호출한다. 

3. Repository (데이터 액세스 계층) : 데이터베이스에 직접 접근하여 데이터를 읽고 쓰는 일만 전문으로 한다. 핵심 기능은 데이터베이스에 접근하기 위한 SQL을 실행하거나 데이터를 조작한다 (Select, Insert, Update, Delete)

4. Entity : 데이터베이스 테이블과 1:1로 직접 매핑되는 클래스 
*주의점 : DB 구조가 그대로 드러나기 때문에, 보안과 안정성을 위해 Controller 까지 절대 노출하지 않아야 함!! 주로 Repository와 Service 사이에서만!! 

5. DTO (Data Transfer Object) : 계층 간 데이터 전송을 위한 것


*나를 위한 정리*
회원가입을 예시로 흐름을 나타내보기
1. Controller : 클라이언트가 준 가입 양식 데이터를 받아서 검증 후 Service로 넘긴다
2. Service : 중복 가입인지 확인하는 로직 실행 후, 문제가 없으면 가입할 회원 데이터를 생성하여 Repository로 넘긴다
3. Repository : 넘어온 Entity를 실제 데이터베이스에 안전하게 저장
4. 저장이 끝나면 저장 결과 OR 데이터를 DTO에 담아 Repository -> Service -> Controller -> 사용자 순으로 응답함




2. 어노테이션 설명
@RestController : 이 클래스가 Controller (프레젠테이션 계층)이고 모든 메서드의 반환 값을 JSON/XML 같은 데이터 형태로 직접 응답하겠다고 선언
없으면? 출력 코드를 일일이 작성해야 함

@Service : 프로그램의 진짜 핵심(기능과 규칙)이 들어있다고 스프링에게 알려줌, 코드 재사용성이 극대화된다
없으면? 싱글톤 패턴을 개발자가 직접 코드로 구현해야한다

@Repository : 데이터베이스와 연결을 시도하는 클래스. 데이터베이스 제품마다 다르게 발생하는 고유의 에러를 스프링이 제공하는 공통 예외로 추상화함
없으면? 데이터베이스 연동 기술을 바꿀 때마다 예외 클래스 명을 코드 전체에서 다 고쳐야함

@Entity : 자바 클래스의 필드 구조를 바탕으로 CREATE TABLE 쿼리를 만들어 데이터베이스에 테이블을 생성하거나 기존 테이블과 연결시킨다
없으면? DB에 테이블을 바꾸면 자바 코드를 바꾸고, 자바 코드를 바꾸면 DB 안 SQL문으로 테이블을 수정하는 이중 작업 필요, 일일이 매핑해야함

@(Get, Post)Mapping : 서버로 들어온 HTTP 요청의 URL 경로와 메서드를 확인하고 일치하는 자바 메서드를 찾아 실행
없으면? 거대한 if-else문이나 switch문을 만들고 값을 대조하면서 분기 처리해야함

@RequestBody : JSON형태의 데이터를 자바 객체의 필드에 매칭하여 변환해준다
없으면? 객체를 수동으로 생성해서 코드를 매번 적어주어야함

@Transactional : 스프링이 해당 클래스나 메서드를 감싸는 가짜 대리 객체 생성, 메서드가 시작될 때 자동으로 DB 커넥션을 연다
없으면? 모든 비즈니스 로직의 코드를 수백 번 도배해야함 

@Data : 자바 코드가 컴파일되어 .class 파일이 바이트코드로 바뀔 때, Lombok 라이브러리가 표준 캡슐화 메서드들을 소스파일 내부에 바이너리 형태로 강제 주입 
없으면? 필드 이름 하나를 바꿀 때마다 전부 수동으로 지우고 다시 생성해야함

*싱글톤 패턴이란 프로그램 전체에서 특정 클래스의 객체를 딱 하나 만들어서 어디서든 공유하며 쓸게라는 규칙 -> 메모리 절약과 성능이 향상됨 


3. CORS
Cross-Origin Resouce Sharing으로 웹 브라우저가 가지는 핵심 보안 기능 중 하나이다. SOP정책을 먼저 알아야한다. SOP는 "내가 접속한 사이트(출처)와 다른 사이트의 데이터는 브라우저가 가져오지 못하게 막는 보안 규칙"이다
업으면? 보안이 취약한 웹사이트에 접속 시 사용자의 권한을 악용하여 개인정보나 메일을 탈취할 수 있음
프론트엔드 서버와 백엔드 API 서버를 따로 개발한다 -> 태생적으로 다른 출처 간에 데이터를 주고받을 수밖에 없다 !!! SOP 생각하면 개발이 안됨...!! 그래서 CORS는 다른 출처의 요청이더라도, 백엔드가 "내가 허용한 안전한 곳이야" 라고 인증해주면 데이터 공유를 허락해주는 정책이다. 
CORS 에러 발생은? 백엔드에서 내는 에러가 아닌 웹 브라우저가 보안 검사를 하다가 차단하는 것이다. 이를 해결하기 위한 방법은 특정 Controller나 메서드에 개별 설정하거나, 글로벌 전역 설정을 이용하는 것이다
요약하자면 CORS는 SOP의 예외 조항이라고 생각하면 될 것 같다!!!
그래서 백엔드 코드에서 프론트엔드의 주소를 허용 주소록에 등록해주면 해결이 된다 !!!


4. 코드 리뷰 
   1. 클라이언트 요청: 사용자가 게시글 내용을 JSON 형식으로 담아 POST /api/boards로 요청을 보냄
   2. Controller (BoardController):
       - @PostMapping 메서드가 요청을 가로채감.
       - @RequestBody BoardRequest를 통해 JSON 데이터가 DTO 객체로 변환됨.
       - boardService.create(request)를 호출함.
   3. Service (BoardService):
       - @Transactional 환경에서 로직이 실행됨.
       - BoardRequest의 데이터를 꺼내 Board 엔티티 객체를 생성.
       - boardRepository.save(board)를 호출하여 DB 저장을 요청.
   4. Repository (BoardRepository):
       - JpaRepository를 상속받은 인터페이스가 실제 DB Insert 쿼리를 실행
   5. 결과 반환:
       - 저장된 엔티티 객체를 서비스에서 다시 응답용 DTO인 BoardResponse로 변환.
       - 컨트롤러를 거쳐 최종적으로 클라이언트에게 JSON 형태로 응답이 전달.

  이 구조는 유지보수와 확장이 용이하며, 각 레이어가 분리되어 있어 역할이 명확한 것이 장점이다!!
